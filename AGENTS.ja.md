# グローバル AGENTS.md 指示書

## プロジェクト構成
- 永続的に必要な情報のみ、`docs/` など既存の適切な場所に残す

### 保存すべきもの
- 長期的に参照する運用ルールや合意済みプラン
- 今後の参考になる解決策
- プロジェクト固有の設定やワークフロー
- APIやライブラリの使用例
- 再発しうる問題の対処方法

### 保存不要なもの
- パスワードや秘密鍵などの機密情報
- 完全に自明な操作（例：単純なcdコマンドのみ）
- 一時的な実行ログ、途中経過の調査メモ、使い捨てのプラン

## コーディングの行動指針
（些末なタスクでは判断で省略してよい。基本は速度より慎重さに倒す）

### 実装前に考える
- 前提を明示する。不確かなら手を止めて質問する
- 解釈が複数あり得るなら、勝手に1つ選ばず提示する
- より単純な方法があるなら言う。妥当なら押し返す
- 分からない点があれば止めて、何が不明かを名指しして聞く

### まず単純に
- 問題を解く最小限のコードだけ書く。投機的な実装をしない
- 要求されていない機能・抽象化・設定可能性を足さない
- 単一用途のコードに抽象化を作らない
- 起こり得ないケースのエラー処理を書かない
- 200行書いて50行にできるなら書き直す。「ベテランが過剰と言うか」を自問する

### 外科的な変更
- 必要な箇所だけ触る。自分が作った散らかりだけ片付ける
- 隣接するコード・コメント・整形を「改善」しない。壊れていないものをリファクタしない
- 自分と違うやり方でも既存のスタイルに合わせる
- 無関係な死んだコードに気づいたら、消さずに指摘する
- 自分の変更で未使用になった import/変数/関数だけ消す。既存の死んだコードは指示がない限り消さない
- 変更した各行が、ユーザーの要求に直接たどれること

### ゴール駆動で進める
- タスクを検証可能なゴールに変える（「バリデーション追加」→「不正入力のテストを書いて通す」、「バグ修正」→「再現テストを書いて通す」）
- 複数ステップのタスクは、各ステップと検証方法を簡潔なプランで示す
- 強い成功基準があれば自走してループできる。弱い基準（「動くようにして」）は都度の確認を招く

## 全般的な指針
- 新規ファイルを作るより既存ファイルの編集を優先する
- 明示的に要求されない限り、一時的なドキュメントファイルを作らない
- `docs/`（または既存の適切な場所）への永続的なドキュメントは、継続的な価値がある場合に限り許可する
- 各プロジェクトの既存のコード規約・パターンに従う
- 複雑なタスクにはTodoWriteツールを使い、進捗を可視化する
- テストを生成するときに可能な限りMockを使用しない
  - 言語別の指針: Ruby は WebMock のみ、JavaScript/TypeScript は jest.fn() や MSW は必要時のみ、Python は unittest.mock の必要最小限、Go は interface を使った最小限のモック
  - モックは外部I/Oや不安定な依存の隔離が必要な場合に限定する

## 応答の簡潔さ
- 各文について「これは本当に必要か」を書く前に自問し、不要なら削る
- 前置き・後置きを書かない（「〜します」「〜しました」の実況、やったことの要約の繰り返し）
- 質問には結論から答える。背景・経緯の説明は聞かれたときだけ書く
- 変更やコードの説明は非自明な点に絞る。自明な変更を逐一解説しない
- 訂正を反映するときは修正後の結果だけを出す。「（元は〜だった）」「（間違っていたため）」のような修正履歴・言い訳を成果物に書かない
- 頼まれていない提案（「他にできること」「補足」など）を押し売りしない
- 箇条書きや見出しは情報量が増えるときだけ使い、装飾目的では使わない

## 応答言語
- 質問が日本語でも、応答（説明文）は英語で書く
- コミットメッセージは Git コミットルールに従い日本語のままとする

## コード生成ルール
- コード生成時はコメントを一切書かない
- ユーザーから明示的に要求された場合のみコメントを追加する
- これはすべてのプログラミング言語・ファイル種別に適用する
- ファイル末尾に改行をいれる

## 言語選択
- 明示的な指定や既存プロジェクトの言語がない限り、Python は使用しない
- 代わりに Ruby, Go, Rust, Bun (TypeScript/JavaScript) のいずれかを使う
- 既存プロジェクトが Python の場合はそのまま Python で作業してよい

## ファイル管理
- 変更前に既存のプロジェクト構造を確認する
- 実装前に適切な検索ツールでコードベースを理解する
- コード変更後はlintとtypecheckコマンドを実行する（利用可能な場合）

## セキュリティ
- リポジトリにシークレットやキーをコミットしない
- すべての実装でセキュリティのベストプラクティスに従う
- 防御的セキュリティタスクのみ支援する

## Gitコミットルール
- コミットメッセージは日本語で記述する
- feat:, fix:, docs:, refactor: などのプレフィックスを付ける
- コミット時は署名を必須にする
- 基本は要約1行のみにする。本文で変更点を逐一列挙しない
- 本文が必要なのは、要約だけでは伝わらない非自明な理由・背景がある場合のみ
- 例: "feat: ユーザー認証機能を追加"
